# State가 길어지면 무슨 문제가 생기나

## Q
state가 너무 길면 뭐가 안 좋은데?

## A
세 가지 문제가 생긴다: **컨텍스트 윈도우 한계**, **비용 증가**, **어텐션 희석**.

---

### 1. 컨텍스트 윈도우 한계

모델은 한 번에 처리할 수 있는 토큰 수에 상한이 있다(예: claude-3.5 sonnet은 200K 토큰).
state는 누적된 메시지 배열이므로, 대화가 길어질수록 토큰 수가 계속 증가한다.
한계를 초과하면 오래된 메시지를 잘라내거나, 호출 자체가 실패한다.

```
state = [msg1, msg2, msg3, ..., msg_n]
         ↑____________________________↑
         이 전체가 매 API 호출마다 전송됨
```

**결론**: state는 모델이 "기억"하는 것처럼 보이지만, 실제로는 매번 전부 다시 전송하는 구조다.

---

### 2. 비용 증가

Anthropic API는 **입력 토큰 수만큼 과금**한다.
state가 길어질수록 매 호출마다 전송되는 입력 토큰이 많아지고, 비용이 선형으로 늘어난다.

예를 들어 루프가 10회 돌았고 state에 10,000토큰이 쌓였다면,
10번째 호출에서 입력만 10,000토큰이다. 1번째 호출(몇백 토큰)과는 비용이 크게 다르다.

---

### 3. 어텐션 희석 (Attention Dilution)

트랜스포머 모델은 입력 전체에 걸쳐 어텐션을 계산한다.
state가 길수록, **현재 작업과 관련 없는 옛날 메시지들**이 어텐션 점수를 일부 가져간다.
중요한 최신 맥락의 가중치가 상대적으로 낮아지고, 모델이 집중해야 할 부분을 놓칠 수 있다.

---

### 이 문제를 해결하는 방법: Compaction

state가 임계값을 넘으면 **요약 호출**을 한 번 더 해서 긴 state를 짧게 압축한다.

```
긴 state → "지금까지의 대화를 요약해줘" 호출 → 짧은 요약 1개로 교체
```

이를 **compaction**이라고 부른다. (Claude Code도 이 방식을 사용한다.)

**트레이드오프**: 요약으로 압축하면 디테일이 손실된다.
"방금 전에 뭐 했지?"를 물었을 때 세부 내용이 사라져 있는 현상이 바로 compaction의 부작용이다.

compaction은 bomini Phase 3에서 직접 구현하며 이 손실을 직접 관찰한다.

---

### 코어 루프와의 관계

CLAUDE.md §1 의 설계도에 따르면, state 관리는 **세션**과 **compaction** 위치에 해당한다:

| 기능 | 코어 루프에서의 위치 |
|---|---|
| 세션 | `state`(누적 메시지 배열)를 저장/복원하는 것 |
| compaction | `state`가 길어지면 요약 호출로 짧게 치환하는 것 |

state의 길이 문제는 "에이전트가 무한히 잘 동작할 수 없는" 현실의 마찰이다.
이 마찰이 버그처럼 보이지만, 사실은 **모델이 API 클라이언트에 불과하다는 사실의 직접적인 결과**다.
